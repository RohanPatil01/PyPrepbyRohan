🧠 Interview Patterns

Think Like an Interviewer

These are the concepts interviewers test again and again. Don't just memorise answers: understand the thinking behind each pattern.

🧠 Patterns
🔒

Mutable vs Immutable

⭐ Most Tested
The Core Idea
Mutable objects can be changed in-place after creation. Immutable objects cannot: any operation creates a new object. This affects aliasing, function arguments, and dictionary keys.
✅ Mutable
list, dict, set, bytearray
Changed in-place. Multiple variables can share and modify the same object.
🔒 Immutable
int, float, str, tuple, frozenset, bool
Cannot be changed. Any "change" creates a new object.
# Mutable - alias problem a = [1, 2, 3] b = a # b is NOT a copy - same object b.append(4) print(a) # [1, 2, 3, 4] ← a was changed! # Immutable - no alias problem x = "hello" y = x y = y + " world" # creates NEW string print(x) # "hello" ← x unchanged # Fix: use copy for lists b = a.copy() # or a[:] or list(a)
🎯 Why interviewers ask this
Interviewers use this to test whether you understand Python's reference model. The classic trap is passing a list to a function and modifying it: the caller's list also changes. Always think: "Is this a new object or the same one?"
🔍

is vs ==

⭐ Most Tested
Identity vs Equality
== checks if two objects have the same value. is checks if two variables point to the exact same object in memory (same id()). Two objects can be equal (==) but not identical (is).
# Lists a = [1, 2] b = [1, 2] print(a == b) # True - same values print(a is b) # False - different objects # Integer caching (-5 to 256) x = 256; y = 256 print(x is y) # True - CPython caches small ints x = 257; y = 257 print(x is y) # False - outside cache range # None - always use 'is' if x is None: # ✅ correct pass if x == None: # ⚠️ works but not idiomatic pass
🎯 Why interviewers ask this
They want to see if you understand Python's memory model. Always use is None / is not None: never == None. Also, never rely on is for integer comparison beyond the -5 to 256 cache: it's an implementation detail.

Variable Scope - LEGB Rule

Very Common
LEGB Lookup Order
When Python resolves a name, it searches in this order: Local → Enclosing → Global → Built-in. The first match wins.
ScopeWhereKeyword to modify
LocalInside current function-
EnclosingOuter function (nested)nonlocal
GlobalModule levelglobal
Built-inPython builtins (len, print…)-
x = 'global' def outer(): x = 'enclosing' def inner(): x = 'local' print(x) # 'local' inner() print(x) # 'enclosing' outer() print(x) # 'global' # Classic gotcha - UnboundLocalError x = 10 def fn(): print(x) # ❌ UnboundLocalError! x = 20 # Python sees this & treats x as local throughout # Fix: def fn(): global x # declare intent to use global x print(x) x = 20
🎯 Why interviewers ask this
The UnboundLocalError trap catches many developers off guard. If Python sees an assignment to a name anywhere in a function, it treats that name as local throughout the entire function: even before the assignment line.
📦

Closures

Common
What Is a Closure?
A closure is an inner function that remembers variables from its enclosing scope even after the outer function has returned. The inner function "closes over" those variables.
def make_multiplier(n): def multiply(x): return x * n # n is captured from enclosing scope return multiply double = make_multiplier(2) triple = make_multiplier(3) print(double(5)) # 10 print(triple(5)) # 15 # Classic loop closure bug fns = [lambda x: x + i for i in range(3)] print(fns[0](10)) # 12 - NOT 10! All lambdas share i=2 # Fix: capture with default argument fns = [lambda x, i=i: x + i for i in range(3)] print(fns[0](10)) # 10 ✅
🎯 Why interviewers ask this
The loop closure bug is one of the most famous Python gotchas. All lambdas in a loop share the same variable: not its value at creation time. The fix using default args i=i captures the value at that moment.
⚠️

Mutable Default Arguments

⭐ Classic Gotcha
The Trap
Default argument values are evaluated once when the function is defined: not on each call. If the default is a mutable object (list, dict), it's shared across all calls that use it.
# ❌ Bug - shared mutable default def add_item(item, lst=[]): lst.append(item) return lst print(add_item(1)) # [1] print(add_item(2)) # [1, 2] ← NOT [2]! print(add_item(3)) # [1, 2, 3] # ✅ Fix - use None as sentinel def add_item(item, lst=None): if lst is None: lst = [] lst.append(item) return lst print(add_item(1)) # [1] ✅ print(add_item(2)) # [2] ✅
🎯 Why interviewers ask this
This is asked in almost every Python interview. The rule is simple: never use a mutable object as a default argument. Always use None and create a fresh object inside the function. Interviewers expect you to explain why this happens at the language level.
📋

Copy vs Deep Copy

Common
Three Levels of Copying
There are three ways variables can relate to an object: assignment (alias), shallow copy, and deep copy. They differ in how nested objects are handled.
MethodOuter containerNested objects
b = aSame objectSame (alias)
b = a.copy() / a[:]New objectShared (same refs)
copy.deepcopy(a)New objectAll new objects
import copy original = [[1, 2], [3, 4]] # Shallow copy shallow = original.copy() shallow[0].append(99) print(original) # [[1, 2, 99], [3, 4]] ← inner list shared! # Deep copy original = [[1, 2], [3, 4]] deep = copy.deepcopy(original) deep[0].append(99) print(original) # [[1, 2], [3, 4]] ← fully independent ✅
🎯 Why interviewers ask this
For flat lists, shallow copy is fine. The danger is with nested structures: shallow copy only duplicates the outer container. Interviewers pair this with the mutability question to test whether you understand shared references.
λ

Lambda Functions

Common
When to Use Lambda
Lambdas are anonymous, single-expression functions. Use them for short, throwaway functions: especially as key arguments to sorted(), min(), max(), or as arguments to map()/filter().
# Good uses of lambda people = [{'name':'Alice','age':30}, {'name':'Bob','age':25}] sorted(people, key=lambda p: p['age']) # Multi-key sort sorted(words, key=lambda w: (len(w), w.lower())) # Conditional expression in lambda classify = lambda x: 'even' if x % 2 == 0 else 'odd' # Lambda vs def - when def is better # ❌ avoid: named lambdas square = lambda x: x**2 # use def instead # ✅ use def for reusable, named functions def square(x): return x**2
🎯 Why interviewers ask this
Interviewers often ask: "What's the difference between lambda and def?" Key answer: lambda is limited to one expression, has no statements, and is anonymous. PEP 8 discourages assigning lambdas to names: use def instead.
🔄

List & Dict Comprehensions

Very Common
Comprehension vs Generator vs Loop
Comprehensions are concise and fast for building collections. Use a generator expression (parentheses) when you only need to iterate once: saves memory. Use a regular loop when logic is complex.
📥

Function Arguments

Common
All Argument Types
Python has five types of function parameters. Order in definition must be: positional → *args → keyword-only → **kwargs.
def fn(pos1, pos2, /, regular, *, kw_only, **kwargs): pass # / separates positional-only (Python 3.8+) # * separates keyword-only after it # *args and **kwargs def demo(*args, **kwargs): print(args) # tuple of positional args print(kwargs) # dict of keyword args demo(1, 2, 3, name='Rohan', age=25) # (1, 2, 3) # {'name': 'Rohan', 'age': 25} # Unpacking on call nums = [1, 2, 3] demo(*nums) # unpack list as positional demo(**{'a':1}) # unpack dict as keyword
⚡

Generators

Medium Priority
Why Generators Matter
Generators produce values lazily: one at a time on demand. They use far less memory than building a full list. A function with yield becomes a generator function.
# Generator function def countdown(n): while n > 0: yield n n -= 1 for x in countdown(3): print(x) # 3, 2, 1 # Generators are exhausted after one pass g = countdown(3) list(g) # [3, 2, 1] list(g) # [] ← exhausted! # next() and StopIteration g = countdown(2) next(g) # 2 next(g) # 1 next(g) # StopIteration raised
🎀

Decorators

Medium Priority
What Is a Decorator?
A decorator is a function that takes another function, wraps it with extra behaviour, and returns the wrapper. The @ syntax is syntactic sugar for fn = decorator(fn).
🎯 Why interviewers ask this
Decorators show mastery of first-class functions and closures. Always use @functools.wraps(func) in your wrapper: without it, the decorated function loses its __name__ and __doc__ metadata.
